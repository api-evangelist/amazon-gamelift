<p>Persistent world use cases have become increasingly popular within multiplayer games, as well as across industries for all kinds of realtime virtual world applications. There are two main games server hosting architectures:&nbsp;session-based&nbsp;and&nbsp;persistent world&nbsp;games.&nbsp;Session-based&nbsp;games group players together through matchmaking for a single short-lived game session.&nbsp;Persistent&nbsp;world&nbsp;games, on the other hand, run the world simulation continuously, with players joining and leaving the world over time. This often requires persisting the state of the world, as well as the state of a player in a specific world.</p> 
<p>On AWS, customers can host persistent world games in multiple ways, including directly on virtual machines on <a href="https://aws.amazon.com/pm/ec2/?trk=36c6da98-7b20-48fa-8225-4784bced9843&amp;sc_channel=ps&amp;s_kwcid=AL!4422!3!467723097970!e!!g!!amazon%20ec2&amp;ef_id=Cj0KCQiA0oagBhDHARIsAI-Bbgeuo2e0ZXhyfzRA0nAOJiI14BBHG7fD-pfNiPdrLDxixglEErEmNTsaAtXcEALw_wcB:G:s&amp;s_kwcid=AL!4422!3!467723097970!e!!g!!amazon%20ec2">Amazon EC2</a>, or containerized on <a href="https://aws.amazon.com/ecs/">Amazon Elastic Container Service</a> or <a href="https://aws.amazon.com/eks/">Amazon Elastic Kubernetes Service</a>. These approaches require you to build your own world orchestration service, as well game session and player session management. They are great options when you want full control of the game session orchestration, and you can handle the operational complexity.</p> 
<p><a href="https://aws.amazon.com/gamelift">Amazon GameLift</a> is a service that deploys and manages dedicated game servers across locations. For customers looking for a more managed solution to reduce the operational effort, it is a great option for both session-based and persistent world games. On Amazon GameLift, you can easily host a single game server fleet globally, where the underlying infrastructure, scaling, and game session management are handled by the service. For session-based games on Amazon GameLift, there are multiple solutions to get started with in the <a href="https://aws.amazon.com/solutions/gametech/game-infrastructure/">AWS Solutions Library</a>. In this post, we’ll focus on the persistent world use case, and we will dive deeper into a new solution guidance: <strong><a href="https://aws.amazon.com/solutions/guidance/persistent-world-game-hosting-on-aws/?did=sl_card&amp;trk=sl_card">Guidance for Persistent World Games on AWS</a></strong>.</p> 
<h3>Solution overview</h3> 
<p>The solution consists of four key components:</p> 
<ul> 
 <li>World manager: a backend service to manage the deployment and status of the worlds</li> 
 <li>Backend APIs: API endpoints for the game client to list and join worlds</li> 
 <li>Amazon GameLift fleet: A multi-region fleet for hosting the game servers across locations</li> 
 <li>Sample game client / server: A Unity sample game client and server implementation</li> 
</ul> 
<h3>World manager</h3> 
<p>The world manager, like all backend components of the solution, is built completely with a serverless architecture to reduce operational complexity. The world manager is an <a href="https://aws.amazon.com/lambda/">AWS Lambda</a> function, which is invoked by <a href="https://aws.amazon.com/eventbridge/">Amazon EventBridge</a> every minute to make sure the worlds configured in the <em>WorldsConfiguration</em> Amazon DynamoDB table are running. The world manager updates the state of the current worlds to another DynamoDB table called <em>WorldSessions</em>. This allows the Backend APIs to access the world information directly from DynamoDB.</p> 
<p>Worlds can be configured with the hosting location, world name, maximum player count, and world map to load. This is an example of the world configuration:</p> 
<p><img alt="Worlds can be configured with the hosting location, world name, maximum player count, and world map to load. This is an example of the world configuration:" class="aligncenter size-full wp-image-4851" height="324" src="https://d2908q01vomqb2.cloudfront.net/91032ad7bbcb6cf72875e8e8207dcfba80173f7c/2023/03/13/image001.jpg" width="1004" /></p> 
<p>Once you have configured the worlds, the world manager Lambda function ensures worlds are running if they are not scheduled for termination. The game server process will receive the world configuration information, including the world map, when the world manager provisions the world using the Amazon GameLift API <em>CreateGameSession</em>. The process then loads the correct map and activates itself to allow player sessions to be created for the world.</p> 
<p><img alt="The process then loads the correct map and activates itself to allow player sessions to be created for the world." class="aligncenter size-full wp-image-4852" height="1170" src="https://d2908q01vomqb2.cloudfront.net/91032ad7bbcb6cf72875e8e8207dcfba80173f7c/2023/03/13/image003.jpg" width="1065" /></p> 
<p>The event flow for the world manager includes the following steps:</p> 
<ol> 
 <li>World manager Lambda function is invoked every 1 minute by Amazon EventBridge. The function checks existing worlds’ status through the Amazon GameLift API</li> 
 <li>World manager then stores the current state of the sessions/worlds to DynamoDB for faster backend access</li> 
 <li>World Manager queries configured worlds from DynamoDB and creates any worlds that are not running by calling the the Amazon GameLift API <em>CreateGameSession</em></li> 
</ol> 
<h3>Dynamic worlds</h3> 
<p>The solution also supports dynamic worlds, where multiple instances of the same world are created as previous are filled with players. This works well for use cases such as lobbies where players gather before joining a session-based game. As the backend is fully customizable, customers can modify the logic of dynamic worlds to allow joining the same world as a friend for example, or returning to the same world after playing a session with friends.</p> 
<p>Dynamic worlds are defined in the <em>WorldsConfiguration</em> table, and the world manager will create new instances of the world when there’s less than one full world of available player sessions left. The dynamic world game server processes will terminate themselves if A) there’s no players for 5 minutes, and B) there’s another copy of the world running (to avoid terminating the final instance of the world). World manager will keep an up to date count of the worlds in the <em>WorldConfiguration</em> table. Here’s an example configuration for dynamic worlds in DynamoDB.</p> 
<p><img alt="Here’s an example configuration for dynamic worlds in DynamoDB." class="aligncenter size-full wp-image-4853" height="210" src="https://d2908q01vomqb2.cloudfront.net/91032ad7bbcb6cf72875e8e8207dcfba80173f7c/2023/03/13/image005.jpg" width="1351" /></p> 
<h3>Backend APIs and the sample client</h3> 
<p>The sample game client is built in Unity, but you can integrate with the Backend API from any game engine. Refer to this blog post on <a href="https://aws.amazon.com/blogs/gametech/how-to-integrate-the-aws-c-sdk-with-unreal-engine/">how to integrate Unreal with AWS SDK</a> as well as <a href="https://docs.aws.amazon.com/gamelift/latest/developerguide/integration-engines-setup-unreal.html">Amazon GameLift on Unreal</a> to build a similar integration in the Unreal engine.</p> 
<p>The client leverages the <a href="https://docs.aws.amazon.com/sdk-for-net/v3/developer-guide/unity-special.html">AWS SDK for .NET with Unity</a> to create a guest identity within an <a href="https://docs.aws.amazon.com/cognito/latest/developerguide/cognito-identity.html">Amazon Cognito Identity Pool</a>, and to then sign requests for <a href="https://aws.amazon.com/api-gateway/">Amazon API Gateway</a> with the credentials of that identity. The supported requests include listing existing worlds within a specific AWS region (Amazon GameLift fleet location), and joining a selected world. The backend APIs access the latest world session information in the WorldSessions table and use the Amazon GameLift API to create player sessions.</p> 
<p><img alt="The event flow for the Backend APIs includes the following steps:" class="aligncenter size-full wp-image-4854" height="1220" src="https://d2908q01vomqb2.cloudfront.net/91032ad7bbcb6cf72875e8e8207dcfba80173f7c/2023/03/13/image007.jpg" width="1458" /></p> 
<p>The event flow for the Backend APIs includes the following steps:</p> 
<ol> 
 <li>Game client requests an identity and credentials from Amazon Cognito Identity Pool to sign API requests</li> 
 <li>Game client requests the world list through Amazon API Gateway. API Gateway triggers a Lambda function that checks game session information in the defined region from DynamoDB.</li> 
 <li>Game client requests to join a specific world in a specific region by calling a Lambda function through API Gateway</li> 
 <li>Lambda function creates a player session for the player, increases the amount of players for that world in DynamoDB, and sends the connection info back to the client</li> 
 <li>Client connects to Amazon GameLift session directly over TCP and sends the player session ID.</li> 
</ol> 
<h3>Amazon GameLift fleet and game server</h3> 
<p>The Amazon GameLift fleet is hosted as a multi-region fleet. The game server build is uploaded only once, and it’s automatically distributed across the fleet locations. Each location also has their own individual scaling, which can follow utilization automatically when you configure <a href="https://docs.aws.amazon.com/gamelift/latest/developerguide/fleets-autoscaling-target.html">Target-based auto scaling</a>.</p> 
<p>The game server process is a headless Unity server build. It validates the player session ID to authenticate the player, and stores player data (only the location in our example) in the <em>WorldPlayerData</em> DynamoDB table. It also polls the <em>WorldsConfiguration</em> table for information on possible requested termination, as well as for the current count of dynamic worlds (if the world is dynamic). <a href="https://aws.amazon.com/cloudwatch/">Amazon CloudWatch</a> is used to collect realtime logs and process-level metrics from the game server. It can also be used to collect custom metrics from the game server process, which is implemented in the sample with a simple StatsD client communicating with the CloudWatch agent.</p> 
<p>The game server runs the full simulation of the game world and has the same world geometry loaded as the clients. Clients only send input to the server. Server runs the simulation and sends the state of the world back to the clients. This is done 30 times per second with the interval reduced as player count grows.</p> 
<p><img alt=" The game server runs the full simulation of the game world and has the same world geometry loaded as the clients. Clients only send input to the server. Server runs the simulation and sends the state of the world back to the clients. This is done 30 times per second with the interval reduced as player count grows." class="aligncenter size-full wp-image-4855" height="966" src="https://d2908q01vomqb2.cloudfront.net/91032ad7bbcb6cf72875e8e8207dcfba80173f7c/2023/03/13/image009.jpg" width="1629" /></p> 
<p>The event flow for the game server includes the following steps:</p> 
<ol> 
 <li>Amazon GameLift session validates the player session ID with the Amazon GameLift server SDK</li> 
 <li>Game server reads world specific player data from DynamoDB and updates the player data as needed. It will store the latest player location after player leaves.</li> 
 <li>Amazon GameLift session checks DynamoDB for scheduled termination and terminates if requested</li> 
 <li>Game server sends logs and metrics to Amazon CloudWatch using the CloudWatch Agent</li> 
</ol> 
<h3>Solution deployment</h3> 
<p>Now that you know the components of the solution, let’s look at how the sample is deployed!</p> 
<h3>Prerequisites</h3> 
<p>To deploy the solution, you are expected to <a href="https://docs.aws.amazon.com/accounts/latest/reference/manage-acct-creating.html">create an AWS account</a>, <a href="https://docs.aws.amazon.com/IAM/latest/UserGuide/id_users_create.html">create an Identity and Access Management (IAM) User</a>, and <a href="https://docs.aws.amazon.com/cli/latest/userguide/cli-configure-files.html">configure AWS credentials</a> on your local system. As the solution leverage the Unity game engine, a basic level of knowledge of Unity is expected as well, though all the steps of using Unity are covered in the deployment instructions.</p> 
<h3>Preliminary setup for the backend and Unity</h3> 
<p>First, <strong>clone</strong> <a href="https://github.com/aws-solutions-library-samples/guidance-for-persistent-world-game-hosting-on-aws">the GitHub repository</a>. After that, <strong>install</strong> the tools for the infrastructure deployment following the <a href="https://github.com/aws-solutions-library-samples/guidance-for-persistent-world-game-hosting-on-aws#preliminary-setup-for-the-backend">Preliminary setup for the backend</a>. This includes <a href="https://aws.amazon.com/cdk/">AWS Cloud Development Kit</a> (AWS CDK) as well as <strong>npm</strong> and <strong>typescript</strong>. AWS CDK is a powerful tool to define and deploy your infrastructure as code, and it’s used to deploy all the components of the solution.</p> 
<p>The sample client and server are built using Unity. <strong>Install</strong> Unity first (the instructions here are for Unity 2020, but newer versions work too, refer to the <a href="https://github.com/aws-solutions-library-samples/guidance-for-persistent-world-game-hosting-on-aws">Readme</a> for details), and then <strong>install</strong> .NET build tools for the build scripts. After installing the scripts, run either <code>./downloadAndSetupUnityDependencies.sh (MacOS/Linux)</code> or <code>downloadAndSetupUnityDependencies.ps1</code> (Windows) to download and build the AWS SDK, Amazon GameLift Server SDK and other dependencies, which are also copied automatically to our Unity project. Refer to <a href="https://github.com/aws-solutions-library-samples/guidance-for-persistent-world-game-hosting-on-aws#preliminary-setup-for-unity">Preliminary setup for Unity</a> for details on configuring your Unity environment and installing the dependencies.</p> 
<h3>Deploying the backend with AWS CDK</h3> 
<p>Now that you have the preliminary setup done, you can start deploying the components! Do this by following the <a href="https://github.com/aws-solutions-library-samples/guidance-for-persistent-world-game-hosting-on-aws#deployment-with-aws-cdk">Deployment with AWS CDK instructions</a>. We will use the default region configuration with home region in <em>US-East-1</em> and additional locations in <em>US-West-2</em> and <em>EU-West-1</em> so you can skip the region configuration steps.</p> 
<p>To install all the dependencies of the AWS CDK app, <strong>run</strong> the following command in the Backend folder:</p> 
<p><code>npm install --force</code></p> 
<p>After that, <strong>run</strong> the following script to bootstrap CDK in your AWS account. The number value is your account ID followed by the deployment region.</p> 
<p><code>cdk bootstrap aws://111122223333/us-east-1</code></p> 
<p>Then, <strong>open</strong> the project in Unity 2020 and build the game server binary. First you need to set the project to game server configuration. <strong>Select</strong> the following menu option:</p> 
<p><em>GameLift → SetAsServerBuild</em></p> 
<p>Then, <strong>select</strong> the following menu option to build the game server:</p> 
<p><em>GameLift → BuildLinuxServer</em></p> 
<p><strong>Select</strong> the <em>LinuxServerBuild</em> folder as the target location.</p> 
<p><img alt="Select the LinuxServerBuild folder as the target location." class="aligncenter size-full wp-image-4856" height="307" src="https://d2908q01vomqb2.cloudfront.net/91032ad7bbcb6cf72875e8e8207dcfba80173f7c/2023/03/13/image011.jpg" width="642" /></p> 
<p>Now you have everything ready for backend deployments. <strong>Run</strong> the following command to synthesize the AWS CDK templates:</p> 
<p><code>cdk synth</code></p> 
<p>Now you’re ready to deploy the first stack that creates the Fleet IAM role (which is used to access DynamoDB and CloudWatch from the game server). <strong>Run</strong> the following command:</p> 
<p><code>cdk deploy PersistentWorldFleetRoleStack</code></p> 
<p>Once that is deployed, <strong>copy</strong> the output for the IAM Role Amazon Resource Name (ARN), and <strong>replace</strong> the CloudWatch agent role_arn value in<code> LinuxServerBuild/amazon-cloudwatch-agent.json</code></p> 
<p>Now you’re ready to deploy the rest of the resources including the world manager, backend APIs and the Amazon GameLift fleet. <strong>Run</strong> the following command:</p> 
<p><code>cdk deploy PersistentWorldGameliftStack</code></p> 
<p>This deployment can take up to 40 minutes as the Amazon GameLift fleet is created across the different locations. Once that is done, you’re ready to set up some worlds!</p>

<h3>Load testing with bot clients on AWS Fargate</h3> 
<p>Now that you have the solution set up end to end with one client, let’s add some load! The guidance comes with a simple load testing configuration that hosts headless bot game clients on <a href="https://aws.amazon.com/fargate/">AWS Fargate</a>. AWS Fargate is a serverless, pay-as-you-go compute engine, and it’s perfect for short term load testing.</p> 
<p>To get started, make sure you have Docker installed and running. Then, configure your AWS account ID in <code>configuration.sh</code> in the <em>UnityBotClient</em> folder. After this, <strong>select</strong> the following menu option to configure the build as a bot client in Unity:</p> 
<p><code>Gamelift → SetAsBotClientBuild</code></p> 
<p>You can now test how the bot operates by running the client. It will pick a random world in a random location and connect to it. The bot will move the player around for several minutes, log out, and connect to another world.</p> 
<p>Next, <strong>select</strong> the following menu options to build the headless bot client:</p> 
<p><code>Gamelift → BuildLinuxBotClient</code></p> 
<p><strong>Select</strong> <em>UnityBotClient</em>/<em>Build</em> folder as the destination for the build.</p> 
<p>Then, <strong>run</strong> the script <code>./buildSetupAndRunBots.sh</code> to deploy the bots to AWS Fargate. It will create the repository for the container image, upload the image, and deploy 10 AWS Fargate Tasks as a Fargate service with 4 clients each, totaling 40 clients. You can easily increase the amount of bots in the Amazon Elastic Container Service console by modifying the service, or by modifying the <code>numberoftasks</code> value in the deployment script and redeploying.</p> 
<p>Now you can log in to the game with a normal client and note the bots moving around the worlds and changing worlds over time. You can also view the Amazon GameLift console, DynamoDB console, as well as API Gateway, Lambda and CloudWatch consoles to view activity, metrics and logs as the bots play the game.</p>
